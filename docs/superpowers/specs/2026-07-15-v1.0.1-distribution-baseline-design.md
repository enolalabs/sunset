# Sunset v1.0.1 Distribution Baseline Design

**Date:** 2026-07-15

**Status:** Approved design

**Target release:** v1.0.1

## 1. Purpose

Sunset v1.0.1 establishes a reliable public distribution baseline. It fixes the repository and Go module identity mismatch, makes GitHub Actions the single release authority, expands native binary coverage, and verifies that every published binary can execute the product's essential workflow.

This is a corrective release. It does not change parsing semantics, output schema, command names, incremental behavior, or dependency-graph behavior. Those changes belong to later phases.

## 2. Program context

The complete improvement program is split into independently designed and implemented phases:

1. **Distribution baseline:** canonical identity, module hygiene, version metadata, and native releases.
2. **Correctness:** transactional incremental state, stale-output cleanup, single-file parsing, and error reporting.
3. **Security and scalability:** symlink containment, collision-free output paths, a bounded worker pool, and memory controls.
4. **Feature completeness:** dependency-graph integration and circular-dependency output.
5. **Delivery quality:** coverage gates, immutable action pinning, signing decisions, and broader release assurance.

This specification covers only Phase 1.

## 3. Decisions

### 3.1 Canonical identity

The canonical repository and Go module path is:

```text
github.com/enolalabs/sunset
```

The project has no external library consumers requiring compatibility with the old singular path. The migration therefore changes the module declaration, all Go imports, documentation, examples, and release configuration directly. No compatibility shim or deprecation module will be created for `github.com/enolalab/sunset`.

### 3.2 Release authority

`.github/workflows/release.yml` is the only release definition. `.goreleaser.yaml` will be removed to eliminate duplicated platform, owner, packaging, and version configuration.

GitHub Actions uses native runners because the tree-sitter bindings require CGO. The release workflow will not rely on cross-compiling CGO artifacts.

### 3.3 Supported targets

v1.0.1 publishes five native targets:

| Operating system | Architecture | GitHub-hosted runner | Executable | Archive format |
|---|---|---|---|---|
| Linux | amd64 | `ubuntu-24.04` | `sunset` | `.tar.gz` |
| Linux | arm64 | `ubuntu-24.04-arm` | `sunset` | `.tar.gz` |
| macOS | amd64 | `macos-15-intel` | `sunset` | `.tar.gz` |
| macOS | arm64 | `macos-15` | `sunset` | `.tar.gz` |
| Windows | amd64 | `windows-2025` | `sunset.exe` | `.zip` |

Archive names follow one rule:

```text
sunset_<version>_<os>_<arch>.<format>
```

For v1.0.1, `<version>` is `1.0.1`, without the leading `v`.

### 3.4 Artifact verification

The release includes one `checksums.txt` containing SHA-256 digests of the five final archives. v1.0.1 does not add Sigstore, Authenticode, Apple code signing, or notarization. Signing and immutable action pinning remain Phase 5 decisions.

## 4. Release architecture

The workflow has three stages:

```text
tag v1.0.1
  -> preflight
  -> five native build-and-smoke-test jobs
  -> aggregate, checksum, and publish
```

### 4.1 Trigger and preflight

The workflow triggers on a pushed semantic-version tag matching `v*`. Preflight must complete before native builds and performs these checks:

1. Parse the tag and require the exact `vX.Y.Z` form.
2. Derive release version `X.Y.Z` once and expose it to downstream jobs.
3. Run `go test ./... -race -count=1`.
4. Run `go vet ./...`.
5. Run `go mod tidy -diff` and fail if it reports a change.
6. Require `go list -m` to equal `github.com/enolalabs/sunset`.
7. Fail if the tracked project contains the old exact path `github.com/enolalab/sunset`.

The workflow never runs a mutating `go mod tidy`. Module metadata must be clean before the tag is created.

### 4.2 Native build jobs

Each target runs on the matching GitHub-hosted runner listed in Section 3.3. These labels are documented as native x64/arm64 runners as of 2026-07-15 in the [GitHub-hosted runners reference](https://docs.github.com/en/actions/reference/runners/github-hosted-runners). If GitHub deprecates a label before implementation, its replacement must preserve the same operating system and CPU architecture; switching to cross-compilation does not satisfy this design.

Each job:

1. Checks out the tagged commit.
2. Sets up the Go version declared in `go.mod`.
3. Enables CGO and builds on the native target.
4. Uses `-trimpath` and release ldflags.
5. Injects the same release version into both:
   - `main.Version`, used by `sunset version`;
   - `github.com/enolalabs/sunset/internal/output.Version`, used in generated project metadata.
6. Runs the target's smoke-test contract.
7. Packages the executable in the target's archive format.
8. Uploads exactly one archive as an internal workflow artifact.

Build jobs need only repository read access. They do not create or modify a GitHub Release.

### 4.3 Smoke-test contract

Every target executes its native binary and verifies:

1. `sunset version` exits successfully and prints the exact tag-derived version.
2. `sunset languages` exits successfully and lists Go, JavaScript, TypeScript, and Python.
3. `sunset parse <fixture> --no-cache --quiet --output <temporary-directory>` exits successfully.
4. The temporary output contains `index.md` and at least one per-file Markdown document.
5. Generated `index.md` contains `sunset_version: <exact-version>`.

The fixture is a small tracked directory, not a single source file, because single-file CLI support is explicitly deferred to Phase 2. Every smoke test uses a fresh temporary output directory and does not reuse a cache.

### 4.4 Aggregate and publish

The aggregate job runs only after all five native jobs succeed. Matrix execution uses `fail-fast: false` so one platform failure does not hide results from the other platforms.

The aggregate job:

1. Downloads all internal workflow artifacts.
2. Requires the exact expected set of five archive names.
3. Rejects missing, duplicate, or unexpected archives.
4. Computes SHA-256 digests from the final archive bytes.
5. Writes the sorted digests to `checksums.txt`.
6. Publishes one GitHub Release named `Sunset v<version>` with generated release notes.
7. Uploads the five archives and `checksums.txt`.

Only the publish job receives `contents: write`. No new repository secret or external signing key is required.

## 5. Failure handling and reproducibility

The release process maintains these invariants:

- A release is never published with only a subset of supported targets.
- Checksums always describe the final downloadable archives, not intermediate executables.
- Version strings, archive names, CLI output, and generated metadata come from the same validated tag.
- Build jobs cannot publish or modify releases.
- Module preparation never mutates the tagged source tree.
- A workflow rerun for the same tag produces the same expected artifact names and updates the corresponding release instead of creating a second release.

Once a public tag has been used for a release, it is immutable. A post-publication defect is corrected with a new patch tag such as `v1.0.2`; the project does not move or overwrite the published tag.

Bit-for-bit reproducibility across different native C toolchains is not a v1.0.1 requirement. The release does require traceable inputs, deterministic naming, native execution, and checksum verification.

## 6. Documentation migration

README and CONTRIBUTING will:

- use `github.com/enolalabs/sunset` for clone, install, imports, and links;
- document all five supported targets;
- show `.zip` handling for Windows and `.tar.gz` handling for Linux/macOS;
- avoid combining the moving `/releases/latest/download/` endpoint with a hard-coded versioned filename;
- show a version-explicit download example;
- retain the existing CLI and library usage contract.

The repository will contain no canonical-installation example using the old singular organization.

## 7. Testing and acceptance criteria

Phase 1 is complete only when all of the following hold:

### Repository checks

- `go test ./... -race -count=1` passes.
- `go vet ./...` passes.
- `go mod tidy -diff` produces no difference.
- `go list -m` returns `github.com/enolalabs/sunset`.
- No tracked source, config, or documentation file contains `github.com/enolalab/sunset`.
- Existing output-schema and snapshot tests remain unchanged unless their only difference is the corrected module identity or version value.

### Workflow checks

- All five targets build on matching native OS/architecture runners.
- All five smoke-test contracts pass.
- A failed target prevents publication.
- The release contains exactly five archives and one checksum file.
- Every archive digest matches `checksums.txt` after downloading release assets.
- `sunset version` and generated `index.md` both report `1.0.1` for the v1.0.1 artifacts.

### Public distribution checks

From a clean Go build and module cache, this command succeeds against the public tag:

```bash
go install github.com/enolalabs/sunset/cmd/sunset@v1.0.1
```

The installed executable passes `sunset version`, `sunset languages`, and the directory-fixture parse smoke test.

## 8. Explicit non-goals

Phase 1 does not:

- repair incremental index/cache/output consistency;
- remove stale per-file output after source deletion;
- add single-file CLI parsing;
- change quiet-mode or per-file error reporting;
- integrate dependency-graph output;
- change the worker-pool architecture;
- add symlink containment or collision-resistant output paths;
- introduce a coverage threshold;
- pin GitHub Actions to immutable commit SHAs;
- sign or notarize release artifacts.

These items remain assigned to Phases 2 through 5 of the improvement program.

## 9. Implementation boundary

Implementation should touch only the files required for canonical identity, dependency metadata, documentation, version injection, release workflow consolidation, and target-specific smoke-test support. It must not opportunistically refactor parser, scanner, cache, engine, analyzer, or output behavior beyond wiring the existing output version variable to release ldflags.
