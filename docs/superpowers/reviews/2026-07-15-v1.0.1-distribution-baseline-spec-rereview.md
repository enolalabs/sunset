**Prior review:** `docs/superpowers/reviews/2026-07-15-v1.0.1-distribution-baseline-spec-review.md`

**Review groups:** Technical and Code Quality, Performance, Security, Process and Operations, Product and People

## Final Verdict: Pass

All 3 Critical and 14 Important findings from the initial review were independently confirmed closed. Re-review found one new Critical and four new Important gaps in the revised acceptance model. The companion spec was corrected for all five, and focused Technical, Performance, and Process validations independently confirmed their closure without new Critical or Important findings.

The specification is ready for implementation planning. Advisory findings remain non-blocking and can be handled during planning or the later delivery-quality phase.

## New findings

### Critical

1. **The final archive is not extracted and executed**
   - **Status:** Closed in the companion spec and confirmed by focused Process and Technical validation.
   - **Sections:** 1.1, 3.3, 4.2–4.5, 7
   - **Issue:** The native job smoke-tests the binary before packaging, while draft verification checks only filenames and digests. Packaging can therefore introduce a wrong root path, lost executable permission, corrupt archive, or different binary without failing the gate.
   - **Required correction:** Package first, extract the final archive with native tooling into a clean directory, validate its one-entry root manifest and POSIX executable permission, and run the complete smoke contract against the extracted executable before upload.

### Important

1. **The public library consumer path lacks an external-module acceptance test**
   - **Status:** Closed in the companion spec and confirmed by focused Technical validation.
   - **Sections:** 1.1, 4.5, 7
   - **Issue:** `go install` validates the command package but does not prove that a new consumer can resolve, import, and compile `pkg/sunset` from the canonical public tag.
   - **Required correction:** Create a temporary external module with clean caches, require the exact tag, import `github.com/enolalabs/sunset/pkg/sunset`, and compile/run a minimal `sunset.Languages()` call.

2. **Verification jobs do not materialize the tracked fixture**
   - **Status:** Closed in the companion spec and confirmed by focused Process validation.
   - **Sections:** 4.1, 4.3, 4.5
   - **Issue:** Jobs have isolated workspaces. Downloading draft assets or installing the module does not place `testdata/go-sample` at the path used by the CLI smoke test.
   - **Required correction:** Checkout the exact tagged commit with pinned checkout and no persisted credentials, then use the fixture's absolute workspace path in draft and published verification-only modes.

3. **The 30-minute performance target has no unique formula**
   - **Status:** Closed in the companion spec and confirmed by focused Performance validation.
   - **Sections:** 4.1, 5.1
   - **Issue:** “Pipeline execution” could mean wall-clock latency, summed job-minutes, or matrix critical-path time, producing inconsistent warnings.
   - **Required correction:** Define the target as active critical-path latency: sequential stage durations plus the longest parallel native job, excluding queue and approval waits. Record total job-minutes separately as cost.

4. **The performance target does not distinguish release-state modes**
   - **Status:** Closed in the companion spec and confirmed by focused Performance validation.
   - **Sections:** 4.1, 5.1
   - **Issue:** Full `absent`, draft-resume, and published verification-only runs execute different stage sets and cannot share a comparable latency baseline.
   - **Required correction:** Apply the 30-minute target only to full `absent` runs. Report other modes separately and exclude them from the full-release baseline.

## Advisory findings

- Publish commit/workflow/toolchain provenance publicly in a later delivery-quality phase.
- Specify per-platform executable-inspection commands during implementation planning.
- Define build-cache policy and report cache hit/miss if release latency becomes unstable.
- Add a small release-operator runbook, draft stale-asset recovery rule, artifact-retention policy, and post-release feedback checklist.
- Monitor aggregate download bytes if the target matrix or artifact size grows.

## Strengths

- The original blocking contradictions are all closed with explicit, testable requirements.
- Draft-to-public promotion, exact-tag state handling, and immutable published releases form a coherent release lifecycle.
- Native CGO execution, exact artifact naming, checksum verification, and least privilege are appropriate for the v1.0.1 scope.
- Runtime support claims are limited to the runner environments actually verified.
- The implementation boundary remains narrow despite stronger distribution tests.
