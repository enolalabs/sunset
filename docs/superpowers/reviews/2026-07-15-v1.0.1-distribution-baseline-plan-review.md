# Plan Review: Sunset v1.0.1 Distribution Baseline

**Date:** 2026-07-15

**Plan:** `docs/superpowers/plans/2026-07-15-v1.0.1-distribution-baseline.md`

**Spec:** `docs/superpowers/specs/2026-07-15-v1.0.1-distribution-baseline-design.md`

**Tech Stack:** Go 1.26.2, CGO/tree-sitter, CLI/public Go library, Bash, PowerShell, GitHub Actions, GitHub Releases, GitHub CLI, native Linux/macOS/Windows runners

**Reviewers:** backend-architect + golang-pro, performance-engineer, security-auditor, devops-engineer, product-manager (specialized prompts executed by general-purpose subagents)

---

## Verdict: Needs Changes

**Summary:** The plan preserves the validated design and covers all three public user paths, but its highest-risk workflow task still leaves native pre-tag validation, mode-dependent job control, release-state revalidation, telemetry, authentication, and several supply-chain assertions to implementer judgment.

**Finding counts:** 0 Critical, 16 Important, 15 Minor

**Spec coverage:** 78 of 82 normalized atomic requirements are fully covered; 3 are partial and 1 is missing.

### Remediation status

The verdict above records the initial plan snapshot. The companion plan was revised to address all 16 Important findings. User-selected resolutions use a separate read-only pre-tag native validation workflow, run the canonical Linux/macOS/Windows installation snippets on matching native validation jobs, and move documentation/release-note inputs before workflow implementation. These corrections remain pending independent re-review before the verdict is considered closed; Minor findings remain advisory unless incorporated into the revised plan.

---

## Strengths

- The repository, governance, and irreversible release work are separated, and Tasks 7–8 require explicit maintainer authorization.
- Identity, shared version resolution, five exact archives, post-extraction smoke tests, draft verification, approval, public verification, and immutable patch recovery trace cleanly to the spec.
- Go version behavior in Task 2 is test-first and covers ldflag override, build-info fallback, CLI output, and generated metadata.
- Official action SHAs were independently confirmed against the current official `v4`/`v5` refs on 2026-07-15.
- The plan serves Go installers, native-binary users, and external Go-library consumers with concrete acceptance paths.
- Platform support and checksum trust boundaries are communicated without claiming untested compatibility or publisher authentication.
- Matrix execution, finite timeouts for most jobs, short-lived artifacts, clean public-install caches, and a non-blocking latency target are appropriate for a small public CLI.
- Phase 1 scope remains disciplined; deferred parser, cache, graph, security/scalability, and signing work is not pulled into this release.

## Findings

### Technical and Code Quality

#### Critical (Must Fix)

None.

#### Important (Should Fix)

1. **Go setup and CGO requirements are incomplete**
   - **Task/Section:** Task 4, Preflight, Native matrix, Draft verification, and Published verification requirements
   - **Issue:** The plan pins `actions/setup-go` but does not require `go-version-file: go.mod` in every Go-running job. It explicitly sets `CGO_ENABLED=1` only on Windows.
   - **Impact:** Hosted-runner defaults may differ from Go 1.26.2, and non-Windows native builds may not enforce the tree-sitter CGO contract.
   - **Recommended fix:** Use the pinned setup-go action with `go-version-file: go.mod` in every job that runs Go, and set/assert `CGO_ENABLED=1` on all five native rows.

2. **Executable helper changes are not test-first**
   - **Task/Section:** Scope rules; Task 3, Steps 1–7
   - **Issue:** Task 3 implements helpers before tests and validates only a Linux happy path plus shell syntax. Failure paths such as wrong version, missing language/output, malformed archive, extra root entry, lost executable permission, and consumer import failure are unspecified.
   - **Impact:** Helpers may accept broken release assets while still satisfying the written steps.
   - **Recommended fix:** Define executable fixture-based tests before helper implementation, covering positive and negative contracts for POSIX and PowerShell helpers.

3. **Release timing is not technically executable as written**
   - **Task/Section:** Task 4, Promotion and public verification; Task 8, Step 7
   - **Issue:** A running `verify_public` job cannot observe its own final `completed_at`. The plan lacks formulas and null/skipped/rerun handling and misstates the 30-minute warning as a target-level miss.
   - **Impact:** Implementations can report inconsistent or incomplete latency and cost metrics.
   - **Recommended fix:** Add a final read-only telemetry job after `verify_public`, define mode-specific formulas and residual wait handling, and correct Task 8 to reference only the absent-mode active-critical-path target.

#### Minor (Nice to Have)

1. Task 6 formats a closed list of Go files; format or verify every changed Go file.
2. Task 3 leaves a release archive in the repository root; use a temporary output directory or clean it before commit.
3. Restate that runner-label or compiler substitution requires reviewed design change and full matrix evidence.
4. Specify exact command signatures, PowerShell parameters, exit codes, output directory, and stdout/GitHub-output contracts for all five helpers.

### Performance

#### Critical (Must Fix)

None.

#### Important (Should Fix)

4. **Preflight lacks its required finite timeout**
   - **Task/Section:** Task 4, Preflight requirements
   - **Issue:** Every other job class has a timeout, but preflight does not require `timeout-minutes: 30`.
   - **Impact:** Race tests, dependency downloads, Git fetches, or API calls may hang beyond the release contract.
   - **Recommended fix:** Add an explicit 30-minute preflight timeout and validate that every job has a finite timeout.

5. **Clean-cache ordering is ambiguous**
   - **Task/Section:** Task 4, Draft and Published verification requirements
   - **Issue:** Fresh cache variables are named, but setup-go may restore default caches or Go commands may run before unique cache paths are exported.
   - **Impact:** The public-install gate may silently reuse cached modules/builds and produce incomparable timing samples.
   - **Recommended fix:** Create/export unique `GOCACHE`, `GOMODCACHE`, and `GOBIN` before any Go command, set setup-go `cache: false`, and state whether the consumer test resets or shares the initially clean caches.

#### Minor (Nice to Have)

5. Replace “such as 3 days” with exact `retention-days: 3`.
6. Put timing calculations in a tested helper with absent/draft/published JSON fixtures.
7. Define timing API pagination and a non-mutating warning/failure policy.

### Security

#### Critical (Must Fix)

None.

#### Important (Should Fix)

6. **Action-pin validation accepts mutable or short refs**
   - **Task/Section:** Task 4, Workflow validation Step 1
   - **Issue:** The regex rejects only `@vN`, `@main`, and `@master`; refs such as `@stable`, branches, and short SHAs pass.
   - **Impact:** A workflow can pass readiness while executing mutable code, including in write-capable jobs.
   - **Recommended fix:** Require each non-local `uses:` entry to match exactly `@[0-9a-fA-F]{40}` and belong to the four-action allowlist; permit only local `./` actions otherwise.

7. **Checkout credential isolation is not workflow-wide**
   - **Task/Section:** Task 4, Native matrix and Published verification requirements
   - **Issue:** `persist-credentials: false` is explicit only for preflight and draft verification.
   - **Impact:** Tagged build/smoke code may read an unnecessary repository token.
   - **Recommended fix:** Make pinned checkout plus `persist-credentials: false` mandatory for every checkout and validate all checkout blocks.

8. **Release-state discovery conflates absence with API failure**
   - **Task/Section:** Task 4, Preflight release query
   - **Issue:** `gh release view` error classification is unspecified.
   - **Impact:** Authentication, rate-limit, network, or parse failure may be treated as `absent` and enter a write path.
   - **Recommended fix:** Treat only authoritative not-found as absent; fail closed on every other error and record release ID, draft flag, tag, and target commit.

9. **Draft mutation relies on stale preflight state**
    - **Task/Section:** Task 4, Draft staging requirements
    - **Issue:** The job deletes/replaces draft assets without immediately revalidating release identity and draft state.
    - **Impact:** A state change between preflight and staging can target a public or different release.
    - **Recommended fix:** Re-query at job start, require no release for absent mode or the recorded draft ID for draft mode, abort on published/mismatch, and mutate assets only by verified release ID.

10. **Approval creates a verification-to-promotion integrity window**
    - **Task/Section:** Task 4, Promotion; Task 8, Steps 5–6
    - **Issue:** The draft is verified before an unbounded human approval wait, but promotion performs no integrity recheck.
    - **Impact:** Draft assets, checksums, notes, target commit, or tag binding can change after verification and before publication.
    - **Recommended fix:** After approval, revalidate the same draft ID, tag binding, target commit, exact assets, sizes/digests, checksum bytes, and notes before promotion.

11. **Tag rules do not explicitly prevent update and deletion**
    - **Task/Section:** Task 7, Step 1; Task 8, Steps 1–2
    - **Issue:** The plan controls tag creation but not update/deletion or repeated exact binding checks.
    - **Impact:** A moved tag can make public module source diverge from reviewed archives.
    - **Recommended fix:** Protect creation, update, and deletion of `v*`; record the peeled annotated-tag commit and assert the binding in preflight, draft verification, and immediately before promotion.

12. **Release-note dependency and safe delivery are undefined**
    - **Task/Section:** Task 4, Draft staging; Task 5, Step 6
    - **Issue:** Task 4 consumes a notes file created by Task 5, and the write-capable job's data path is unspecified.
    - **Impact:** Task 4 is not independently complete; unsafe shell interpolation or an unnecessary credentialed checkout could be introduced.
    - **Recommended fix:** Create notes before workflow implementation, retrieve them from the exact commit through a read-only path, and pass them to `gh release create --notes-file` without shell interpolation.

#### Minor (Nice to Have)

8. Record `gh version` with runner/toolchain evidence.
9. Add `go mod verify`; consider a pinned non-blocking vulnerability check if kept within Phase 1.
10. Make `actionlint@v1.7.7` reproducible through a declared Go tool dependency or documented sumdb verification.
11. Require public asset downloads without `Authorization` or `GH_TOKEN`.

### Process and Operations

#### Critical (Must Fix)

None.

#### Important (Should Fix)

13. **No read-only native validation runs before the immutable release tag**
    - **Task/Section:** Task 3, Step 7; Task 4 trigger; Task 6; Task 8, Steps 2–3
    - **Issue:** Windows, macOS, ARM, MinGW, PowerShell, runner labels, and matrix expressions first execute after `v1.0.1` is pushed.
    - **Impact:** A workflow defect can burn the immutable tag before a valid draft exists.
    - **Recommended fix:** Add a pre-tag read-only validation path for an explicit protected-main commit that runs preflight plus all five native package/extract/smoke jobs and becomes a Task 7/8 prerequisite.

14. **Mode-dependent workflow DAG has no exact `needs`/`if` truth table**
    - **Task/Section:** Task 4, Workflow structure
    - **Issue:** GitHub skip propagation can suppress published-mode verification, while careless `always()` can run after failed prerequisites.
    - **Impact:** Required verification may not run, or may run after an invalid upstream state.
    - **Recommended fix:** Specify exact dependencies and conditions. `verify_public` must require successful preflight and either published mode or successful promotion for absent/draft mode. Test all modes and upstream-failure cases.

15. **GitHub CLI authentication is not specified**
    - **Task/Section:** Task 4, every `gh` invocation
    - **Issue:** Job permissions do not automatically configure GitHub CLI authentication.
    - **Impact:** Release queries, uploads, downloads, promotion, and timing calls fail despite correct permissions.
    - **Recommended fix:** Set step-scoped `GH_TOKEN: ${{ github.token }}` only for `gh` steps with matching minimum permissions; never expose it to build or smoke commands.

#### Minor (Nice to Have)

12. Resolve and record Windows `objdump` before architecture inspection.
13. Define draft cleanup/upload failure ordering and recoverable rerun behavior.

### Product and People

#### Critical (Must Fix)

None.

#### Important (Should Fix)

16. **Cross-platform copy-paste documentation is not acceptance-tested**
    - **Task/Section:** Task 5, Steps 2–4 and 8; Task 8, Step 8
    - **Issue:** The plan documents Linux, macOS, and Windows commands but manually verifies only one suitable archive and does not execute the exact published commands.
    - **Impact:** The corrective release can repeat its original failure mode on an untested platform.
    - **Recommended fix:** Run the exact documented download, checksum, extraction, and version commands for Linux, macOS, and Windows on matching native jobs or as explicit operator scenarios sourced from the published documentation.

#### Minor (Nice to Have)

14. Define a lightweight post-release install/platform feedback checkpoint.
15. Specify the link-checking method and exact-tag asset/checksum URLs.

## Recommendations

1. Add a read-only pre-tag native validation design before any release tag can be created.
2. Make Task 4 executable through exact job DAG conditions, state revalidation, scoped GitHub authentication, and a separate telemetry job.
3. Turn release helper and workflow-policy assertions into deterministic fixture-based tests.
4. Move curated release notes ahead of workflow staging and pass them only as a safely quoted file.
5. Re-run all five review groups after fixes because the findings touch workflow control flow, supply-chain policy, native validation, and user onboarding.
