# Spec Review: Sunset v1.0.1 Distribution Baseline

**Date:** 2026-07-15

**Spec:** `docs/superpowers/specs/2026-07-15-v1.0.1-distribution-baseline-design.md`

**Tech Stack:** Go 1.26.2, CGO, tree-sitter, GitHub Actions, GitHub Releases, native Linux/macOS/Windows runners

**Reviewers:** backend-architect + golang-pro, performance-engineer, security-auditor, devops-engineer, product-manager (specialized prompts executed by general-purpose subagents)

---

## Verdict: Needs Changes

**Summary:** The native release architecture, scope boundaries, and executable acceptance model are strong. The spec is not implementation-ready because three requirements cannot currently pass and the public-release promotion and immutability policies leave material supply-chain and user-trust ambiguity.

**Unique finding counts:** 3 Critical, 14 Important, 15 Minor

### Remediation status

The verdict above records the initial review snapshot. The companion spec was updated after review to address all 3 Critical and all 14 Important findings. The independent re-review confirmed that every original Critical and Important finding is closed; new findings and their remediation are recorded in `docs/superpowers/reviews/2026-07-15-v1.0.1-distribution-baseline-spec-rereview.md`. The original Minor findings remain advisory unless explicitly incorporated into the revised design.

## Findings

### Technical and Code Quality

#### Critical (Must Fix)

1. **Public `go install` cannot satisfy the version contract**
   - **Section:** 4.2, 4.5, 5, and 7
   - **Issue:** Packaged archives receive version ldflags, but `go install github.com/enolalabs/sunset/cmd/sunset@v1.0.1` does not. The installed binary therefore retains `dev` for CLI and generated metadata.
   - **Impact:** A mandatory public-distribution acceptance path is structurally incapable of passing.
   - **Recommended fix:** Define one runtime version source shared by CLI and output metadata. It accepts an ldflag override for packaged binaries and otherwise derives the module version from `runtime/debug.ReadBuildInfo()`, normalizing the leading `v`.

2. **The specified parse smoke command uses unsupported flag order**
   - **Section:** 4.3 and 4.5
   - **Issue:** The spec places `<fixture>` before flags. The standard Go `flag.FlagSet` used by the CLI stops at the first positional argument, so the later flags are ignored.
   - **Impact:** The smoke test writes to the default output location and fails its own assertions on every platform.
   - **Recommended fix:** Specify `sunset parse --no-cache --quiet --output <temporary-directory> <fixture>` everywhere.

3. **The canonical-identity check rejects the spec itself**
   - **Section:** 3.1, 4.1 item 7, 6, and 7
   - **Issue:** The spec and review history necessarily contain the old path while preflight rejects that string in all tracked documentation.
   - **Impact:** Preflight can never pass without deleting legitimate design and audit history.
   - **Recommended fix:** Scope the check to active canonical/runtime surfaces: `go.mod`, Go source, active workflows/configuration, README, CONTRIBUTING, and current user-facing examples. Explicitly exclude `docs/superpowers/**` and audit/review history.

#### Important (Should Fix)

4. **Exact archive identifiers are underspecified**
   - **Section:** 3.3 and 4.4
   - **Issue:** `<os>` does not define whether macOS archives use `darwin` or `macos`, yet aggregate validation requires an exact set.
   - **Impact:** Workflow, documentation, and release URLs can disagree.
   - **Recommended fix:** Enumerate all five exact v1.0.1 filenames and define canonical OS tokens as `linux`, `darwin`, and `windows`.

5. **Native host architecture is not asserted**
   - **Section:** 3.3 and 4.2
   - **Issue:** Runner labels imply architecture, but jobs do not assert `GOHOSTOS`/`GOHOSTARCH` or inspect the built executable.
   - **Impact:** A runner-label/image change could silently publish a mislabeled architecture.
   - **Recommended fix:** Require host OS/architecture assertions before build and executable architecture inspection where platform tooling supports it.

#### Minor (Nice to Have)

6. **Semantic tag grammar is descriptive, not executable**
   - **Section:** 4.1
   - **Recommendation:** Specify the accepted regex and leading-zero/prerelease policy.

7. **Fixture contract has no stable path or minimal contents**
   - **Section:** 4.3
   - **Recommendation:** Name the tracked fixture and require deterministic supported source with no external dependency.

### Performance

#### Critical (Must Fix)

None.

#### Important (Should Fix)

8. **The release pipeline has no duration target or stage telemetry**
   - **Section:** 4 and 7
   - **Issue:** Correctness gates exist, but the spec has no expected total duration or stage-level timing visibility.
   - **Impact:** A 2–10x release-time regression can pass every acceptance criterion.
   - **Recommended fix:** Record each stage and total workflow duration. Set an initial non-blocking operational target, such as completing within 30 minutes under normal hosted-runner conditions, and revise it from observed runs.

9. **Network and CGO jobs have no finite timeout**
   - **Section:** 4.1, 4.2, 4.5, and 5
   - **Issue:** Dependency downloads, direct module installation, and native compilation can hang without a job timeout.
   - **Impact:** One failure can occupy scarce native runners for hours, especially with `fail-fast: false`.
   - **Recommended fix:** Define finite job timeouts, such as 30 minutes for preflight/build jobs and 20 minutes for distribution verification.

#### Minor (Nice to Have)

10. **Build/dependency cache policy is unspecified**
    - **Section:** 4.1 and 4.2
    - **Recommendation:** Permit setup-go module/build caching keyed by OS, architecture, and `go.sum`; explicitly disable reuse for the clean public-install check.

11. **Same-tag workflow concurrency is unspecified**
    - **Section:** 4.1 and 5
    - **Recommendation:** Use a concurrency group keyed by tag and do not cancel workflows for different tags.

12. **Public verification downloads every archive on one runner**
    - **Section:** 4.5
    - **Recommendation:** Keep this for five small v1.0.1 assets, but monitor total download size if the matrix grows.

### Security

#### Critical (Must Fix)

None beyond the cross-group blocking findings above.

#### Important (Should Fix)

13. **Release actions remain mutable while holding write permission**
    - **Section:** 3.4, 4.2, 4.4, and 8
    - **Issue:** Major-version action tags can move. A compromised action in the publish job executes with `contents: write`.
    - **Impact:** An attacker can replace artifacts/checksums or mutate the public release.
    - **Recommended fix:** Pin every release-workflow action to a full commit SHA in Phase 1, retaining an upstream-version comment for dependency updates.

14. **Verification happens after assets become public**
    - **Section:** 4, 4.4, and 4.5
    - **Issue:** A packaging or integrity failure is visible to users before the final job reports failure.
    - **Impact:** Users can download a release that does not meet its own acceptance criteria.
    - **Recommended fix:** Create a draft release, upload and verify the exact draft assets, verify the public module tag, then promote the draft to public. A lightweight post-public download check may remain non-gating.

15. **Published-asset rerun semantics violate immutability**
    - **Section:** 5
    - **Issue:** The spec allows reruns to update an existing release despite requiring defects to use a new patch tag.
    - **Impact:** The same version can serve different bytes and checksums at different times.
    - **Recommended fix:** Allow replacement only while a release is draft. After publication, reruns verify existing assets without mutation or fail and require a new patch tag.

16. **Release-tag provenance and promotion authority are undefined**
    - **Section:** 4.1 and 5
    - **Issue:** Any permitted `v*` tag push can publish a release without proving the commit belongs to the protected release branch or receiving promotion approval.
    - **Impact:** A typo, compromised maintainer, or automation can publish an unintended commit.
    - **Recommended fix:** Protect `v*` tags with a GitHub ruleset, require tagged commits to be reachable from the protected default branch, and gate publication through a protected `release` environment with required maintainer approval.

#### Minor (Nice to Have)

17. **Checksum trust limits are not explained**
    - **Section:** 3.4 and 6
    - **Recommendation:** State that SHA-256 detects corruption but does not authenticate the publisher; signing remains Phase 5.

18. **Checkout credential persistence is not disabled explicitly**
    - **Section:** 4.2 and 4.4
    - **Recommendation:** Use `persist-credentials: false` and avoid checkout entirely in aggregation/publish where unnecessary.

19. **Release provenance is not surfaced**
    - **Section:** 5
    - **Recommendation:** Include commit SHA, workflow URL, Go version, runner image, and compiler version in release metadata or a manifest.

### Process and Operations

#### Critical (Must Fix)

The impossible identity scan is Critical finding 3.

#### Important (Should Fix)

20. **Windows CGO toolchain is not operationally specified**
    - **Section:** 3.2, 3.3, and 4.2
    - **Issue:** Enabling CGO does not identify a supported compiler/linker or verify it is present on `windows-2025`.
    - **Impact:** Windows builds may fail or silently change characteristics as runner images evolve.
    - **Recommended fix:** Specify the Windows C toolchain and setup source, verify compiler identity/version, and run a minimal CGO/tree-sitter build check before packaging.

21. **Runner replacement has no reviewed change gate**
    - **Section:** 4.2
    - **Issue:** The spec permits label replacement based only on OS/architecture equivalence.
    - **Impact:** SDK, compiler, minimum OS, and packaging behavior can change without review.
    - **Recommended fix:** Treat label/toolchain replacement as a reviewed workflow change and record selected runner/toolchain versions in release evidence.

#### Minor (Nice to Have)

22. **No release-operator runbook is identified**
    - **Section:** 5 and 6
    - **Recommendation:** Add a short runbook covering tag creation, approval, expected assets, rerun rules, verification, and patch recovery.

23. **Intermediate artifact retention is unspecified**
    - **Section:** 4.2 and 4.4
    - **Recommendation:** Set a short explicit retention period and name workflow artifacts by tag plus target.

24. **Maintainer-facing release evidence is incomplete**
    - **Section:** 4.2–4.5
    - **Recommendation:** Write a job summary with tag, commit, runner image, Go/C compiler versions, target, test result, filenames, and checksums.

25. **Transient GitHub failure recovery is unstated**
    - **Section:** 4.4 and 4.5
    - **Recommendation:** Allow rerun only while no published asset would be overwritten; let the immutability rule decide whether recovery proceeds.

### Product and People

#### Critical (Must Fix)

None.

#### Important (Should Fix)

26. **User groups and their success paths are implicit**
    - **Section:** 1, 3, 6, and 7
    - **Issue:** The spec does not explicitly distinguish Go installers, binary-download users, and Go-library consumers.
    - **Impact:** Validation and documentation can cover one path while leaving another confusing or broken.
    - **Recommended fix:** Add an audience/use-case section with measurable success for all three groups.

27. **Runtime compatibility is undefined for supported targets**
    - **Section:** 3.3 and 7
    - **Issue:** Native runner success does not define minimum OS, libc, or runtime compatibility.
    - **Impact:** Users can interpret “Linux amd64” or “macOS amd64” more broadly than the tested environment.
    - **Recommended fix:** Either define and test minimum runtime baselines or phrase the contract as “built and verified on” the named runner environments.

28. **Canonical-path migration lacks curated release communication**
    - **Section:** 3.1, 4.4, and 6
    - **Issue:** Generated notes alone do not explain the corrected install/import path and lack of compatibility shim.
    - **Impact:** Users may continue following the broken singular path.
    - **Recommended fix:** Require curated release-note content covering the canonical install/import path, new platforms, checksum usage, and compatibility decision.

29. **Checksum verification instructions are not acceptance criteria**
    - **Section:** 3.4 and 6
    - **Issue:** Checksums are published without requiring usable POSIX and Windows verification examples.
    - **Impact:** Many users cannot benefit from the integrity mechanism.
    - **Recommended fix:** Require documented checksum verification commands for POSIX and Windows.

#### Minor (Nice to Have)

30. **Archive contents are not part of the user contract**
    - **Section:** 3.3 and 4.2
    - **Recommendation:** Define whether archives contain only the executable or also README/LICENSE, and specify the executable path after extraction.

31. **No small post-release product feedback loop is defined**
    - **Section:** 7
    - **Recommendation:** Confirm docs links and record installation/platform issues before the next patch; formal analytics are unnecessary.

32. **The no-consumer assumption lacks audit context**
    - **Section:** 3.1
    - **Recommendation:** Record it explicitly as an accepted product assumption from the design discussion.

---

## Strengths

- The spec isolates a corrective release from later correctness, security/scalability, feature, and delivery phases.
- Native OS/architecture execution is the right default for CGO/tree-sitter and avoids unverified cross-toolchains.
- Preflight, build, aggregate, and verification responsibilities are clearly separated and testable.
- Least privilege is present at the job level, with write access intended only for publication.
- Artifact formats, checksum generation, smoke tests, and public installation are measurable rather than aspirational.
- Explicit non-goals protect the implementation from unrelated parser/engine refactoring.

## Recommendations

Fix the three blocking contradictions first. Then define a draft-to-public promotion state machine with immutable published assets; the promotion model resolves findings 14, 15, and much of 16 in one coherent design. Complete the operational contract with exact artifact names, native host/toolchain assertions, finite timeouts, and user-facing migration/checksum documentation.
