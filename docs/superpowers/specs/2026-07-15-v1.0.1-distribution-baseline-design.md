# Sunset v1.0.1 Distribution Baseline Design

**Date:** 2026-07-15

**Status:** Approved design

**Target release:** v1.0.1

## 1. Purpose

Sunset v1.0.1 establishes a reliable public distribution baseline. It fixes the repository and Go module identity mismatch, makes GitHub Actions the single release authority, expands native binary coverage, and verifies that every published binary can execute the product's essential workflow.

This is a corrective release. It does not change parsing semantics, output schema, command names, incremental behavior, or dependency-graph behavior. Those changes belong to later phases.

### 1.1 Audiences and success paths

Phase 1 serves three public audiences:

- **Go installers:** `go install github.com/enolalabs/sunset/cmd/sunset@v1.0.1` installs a binary that reports the tagged version and completes the directory smoke test.
- **Prebuilt-binary users:** users can select the archive for one of the five verified targets, validate its SHA-256 digest using documented POSIX or Windows commands, extract it, and run the same smoke contract against the extracted executable.
- **Go library consumers:** a new external Go module can require the exact public tag, import `github.com/enolalabs/sunset/pkg/sunset`, compile, and call the public API without encountering a module-path mismatch.

The direct module migration assumes there are no external library consumers requiring compatibility with the old path. This is an accepted product assumption from the 2026-07-15 design discussion, not a compatibility guarantee discovered through telemetry.

## 2. Program context

The complete improvement program is split into independently designed and implemented phases:

1. **Distribution baseline:** canonical identity, module hygiene, version metadata, and native releases.
2. **Correctness:** transactional incremental state, stale-output cleanup, single-file parsing, and error reporting.
3. **Security and scalability:** symlink containment, collision-free output paths, a bounded worker pool, and memory controls.
4. **Feature completeness:** dependency-graph integration and circular-dependency output.
5. **Delivery quality:** coverage gates, signing decisions, attestations, and broader release assurance.

This specification covers only Phase 1.

## 3. Decisions

### 3.1 Canonical identity

The canonical repository and Go module path is:

```text
github.com/enolalabs/sunset
```

The project has no external library consumers requiring compatibility with the old singular path. The migration therefore changes the module declaration, all Go imports, documentation, examples, and release configuration directly. No compatibility shim or deprecation module will be created for `github.com/enolalab/sunset`.

### 3.2 Release authority

`.github/workflows/release.yml` is the only release definition. `.goreleaser.yaml` will be removed to eliminate duplicated platform, owner, packaging, and version configuration.

GitHub Actions uses native runners because the tree-sitter bindings require CGO. The release workflow will not rely on cross-compiling CGO artifacts.

Every action used by the release workflow is pinned to a full commit SHA in Phase 1. A comment records the corresponding upstream release version so automated dependency tooling can propose reviewed updates. Checkout uses `persist-credentials: false`, and aggregation/promotion jobs do not checkout the repository unless a documented step requires it.

### 3.3 Verified distribution targets

v1.0.1 publishes five native targets built and verified on the named GitHub-hosted runner environments:

| Operating system | Architecture | GitHub-hosted runner | Executable | Exact v1.0.1 archive |
|---|---|---|---|---|
| Linux | amd64 | `ubuntu-24.04` | `sunset` | `sunset_1.0.1_linux_amd64.tar.gz` |
| Linux | arm64 | `ubuntu-24.04-arm` | `sunset` | `sunset_1.0.1_linux_arm64.tar.gz` |
| macOS | amd64 | `macos-15-intel` | `sunset` | `sunset_1.0.1_darwin_amd64.tar.gz` |
| macOS | arm64 | `macos-15` | `sunset` | `sunset_1.0.1_darwin_arm64.tar.gz` |
| Windows | amd64 | `windows-2025` | `sunset.exe` | `sunset_1.0.1_windows_amd64.zip` |

Archive names follow one rule:

```text
sunset_<version>_<os>_<arch>.<format>
```

For v1.0.1, `<version>` is `1.0.1`, without the leading `v`.

Canonical filename OS tokens are `linux`, `darwin`, and `windows`. Every archive contains only the executable at its archive root (`sunset` or `sunset.exe`); documentation and license remain separate release/repository content.

For v1.0.1, “verified” means the artifact built and passed the smoke contract on the exact runner environment in the table. It is not a claim of compatibility with every older distribution, libc, macOS release, or Windows release sharing the same OS/architecture label. Establishing and testing minimum runtime baselines requires a separate compatibility design.

### 3.4 Shared version resolution

CLI output and generated project metadata use one internal version resolver rather than independent package variables. The resolver follows this precedence:

1. Return a non-empty build override injected through `-ldflags` for packaged release binaries.
2. Otherwise read the main module version from `runtime/debug.ReadBuildInfo()` for binaries installed with `go install <module>@<version>`.
3. Normalize a leading `v`, so module version `v1.0.1` is displayed and emitted as `1.0.1`.
4. Fall back to `dev` when build information is absent or reports `(devel)`.

The build override is a string variable in the shared internal version package, for example:

```text
-X github.com/enolalabs/sunset/internal/version.BuildVersion=1.0.1
```

Both `sunset version` and `sunset_version` in generated `index.md` call this resolver. The public-install acceptance path therefore reports the tagged module version without requiring custom consumer ldflags.

### 3.5 Artifact verification

The release includes one `checksums.txt` containing SHA-256 digests of the five final archives. Checksums detect corruption and byte mismatch; they do not authenticate the publisher if an attacker controls both archives and checksum metadata. v1.0.1 does not add Sigstore, Authenticode, Apple code signing, or notarization. Signing and attestations remain Phase 5 decisions.

## 4. Release architecture

The workflow has six stages:

```text
tag v1.0.1
  -> preflight
  -> five native build-and-smoke-test jobs
  -> aggregate, checksum, and draft upload
  -> draft-asset and public-module verification
  -> promote draft to public release
  -> post-public download verification
```

### 4.1 Trigger and preflight

The workflow triggers on a pushed semantic-version tag matching `v*`. Preflight must complete before native builds and performs these checks:

1. Parse the tag and require `^v(0|[1-9][0-9]*)\.(0|[1-9][0-9]*)\.(0|[1-9][0-9]*)$`; prerelease/build suffixes and zero-padded components are rejected.
2. Derive release version `X.Y.Z` once and expose it to downstream jobs.
3. Run `go test ./... -race -count=1`.
4. Run `go vet ./...`.
5. Run `go mod tidy -diff` and fail if it reports a change.
6. Require `go list -m` to equal `github.com/enolalabs/sunset`.
7. Fail if active runtime, release, or user-facing surfaces contain the old exact path `github.com/enolalab/sunset`.
8. Require the tagged commit to be reachable from the protected `main` branch.

The workflow never runs a mutating `go mod tidy`. Module metadata must be clean before the tag is created.

The identity scan covers all tracked `*.go` files, `go.mod`, active files under `.github/workflows/`, README, CONTRIBUTING, and the Makefile. Historical design, review, and audit records under `docs/superpowers/**` and `PROJECT_AUDIT_REPORT.md` are explicitly outside this invariant because they must retain an accurate record of the migration.

Preflight also queries the exact-tag release state and exposes one of three modes:

- `absent`: run the full build, draft, verify, and promotion flow;
- `draft`: run the full build and verification flow, replacing only assets on the non-public draft before promotion;
- `published`: skip all build and write-capable jobs, verify the existing public asset set/checksums and public module installation, and never mutate the release.

Workflow concurrency is keyed by the exact tag with cancellation disabled. Runs for different tags do not block one another, while two runs for the same tag cannot stage or promote concurrently.

Repository governance protects the `v*` tag namespace with a GitHub ruleset that permits tag creation only for designated release maintainers or release automation. The promotion job targets a protected GitHub environment named `release` and requires approval from a release maintainer after draft verification succeeds. Preflight, native builds, draft staging, and verification run before this approval gate; only the final transition to public waits for approval.

### 4.2 Native build jobs

Each target runs on the matching GitHub-hosted runner listed in Section 3.3. These labels are documented as native x64/arm64 runners as of 2026-07-15 in the [GitHub-hosted runners reference](https://docs.github.com/en/actions/reference/runners/github-hosted-runners). If GitHub deprecates a label before implementation, replacement is a reviewed workflow change that must preserve the same operating system and CPU architecture, run the full matrix contract, and record the new runner/toolchain evidence. Switching to cross-compilation does not satisfy this design.

Each job:

1. Checks out the tagged commit.
2. Sets up the Go version declared in `go.mod`.
3. Asserts `go env GOHOSTOS GOHOSTARCH` equals the matrix OS/architecture before build.
4. Enables CGO, records the runner image and C compiler identity/version, and builds on the native target.
5. Uses `-trimpath` and release ldflags.
6. Injects the release version into `github.com/enolalabs/sunset/internal/version.BuildVersion`; both CLI and output metadata resolve through that package.
7. Executes the built binary and, where standard platform tooling supports it, inspects the executable format/architecture before packaging.
8. Packages the executable in the target's archive format.
9. Extracts that final archive with native tooling into a fresh temporary directory and requires exactly one root entry with the target's expected executable name. Linux and macOS also require the extracted file to retain executable permission.
10. Runs the complete target smoke-test contract against the extracted executable, not the pre-packaging path.
11. Uploads only the archive that passed the post-extraction smoke test as an internal workflow artifact named with the tag and target.

Build jobs need only repository read access. They do not create or modify a GitHub Release.

#### Windows CGO toolchain

The `windows-2025` job uses the MinGW-w64 GCC toolchain preinstalled on the GitHub-hosted runner image; it does not install a second MSYS2/MinGW distribution. Before building, PowerShell must:

1. Resolve `gcc` with `Get-Command gcc` and fail if no native executable is found.
2. Record the resolved compiler path, `gcc --version`, and runner image metadata in the job summary.
3. Set `CC` to the resolved GCC executable and require `CGO_ENABLED=1`.
4. Run the normal Go build plus tree-sitter smoke contract; successful native execution is the compatibility gate for the C compiler/linker combination.

If the hosted image removes or materially changes this toolchain, adding or replacing a compiler is a reviewed workflow-design change under the runner replacement policy rather than an implicit package installation during release.

### 4.3 Smoke-test contract

Every target executes its native binary and verifies:

1. `sunset version` exits successfully and prints the exact tag-derived version.
2. `sunset languages` exits successfully and lists Go, JavaScript, TypeScript, and Python.
3. `sunset parse --no-cache --quiet --output <temporary-directory> <fixture>` exits successfully. Flags precede the positional fixture because the CLI uses Go's standard `flag.FlagSet`.
4. The temporary output contains `index.md` and at least one per-file Markdown document.
5. Generated `index.md` contains `sunset_version: <exact-version>`.

The fixture is the tracked `testdata/go-sample` directory, which contains deterministic supported source and has no external runtime dependency. It is a directory rather than a single source file because single-file CLI support is explicitly deferred to Phase 2. Every smoke test uses a fresh temporary output directory and does not reuse a cache.

### 4.4 Aggregate and stage a draft release

The aggregate job runs only after all five native jobs succeed. Matrix execution uses `fail-fast: false` so one platform failure does not hide results from the other platforms.

The aggregate job:

1. Downloads all internal workflow artifacts.
2. Requires the exact expected set of five archive names.
3. Rejects missing, duplicate, or unexpected archives.
4. Computes SHA-256 digests from the final archive bytes.
5. Writes the sorted digests to `checksums.txt`.
6. Creates or resumes one draft GitHub Release named `Sunset v<version>`.
7. Adds curated release notes that state the corrected canonical install/import path, absence of a compatibility shim, five supported targets, archive formats, and checksum-verification links.
8. Uploads the five archives and `checksums.txt` to the draft.

The draft remains non-public until the verification stage succeeds. Only draft-staging and promotion jobs receive `contents: write`. No new repository secret or external signing key is required.

### 4.5 Draft-asset and public-module verification

A verification job runs on `ubuntu-24.04` before publication. It validates the bytes uploaded to the draft rather than reusing internal workflow artifacts:

1. Checkout the exact tagged commit with the full-SHA-pinned checkout action and `persist-credentials: false`. Resolve `testdata/go-sample` to an absolute workspace path; no other job workspace is assumed to exist.
2. Download the draft release assets for the exact tag through the GitHub API into an empty directory.
3. Require exactly the five archives and `checksums.txt`.
4. Run SHA-256 verification for every downloaded archive.
5. Create fresh `GOCACHE`, `GOMODCACHE`, and `GOBIN` directories.
6. Install the public module directly from the exact tag with `GOPROXY=direct`:

   ```bash
   go install github.com/enolalabs/sunset/cmd/sunset@v1.0.1
   ```

7. Run `version`, `languages`, and `parse --no-cache --quiet --output <temporary-directory> <absolute-fixture-path>` against the installed executable. Require both CLI and generated metadata to report the exact module version.
8. In a fresh directory outside the checkout, initialize a temporary consumer module, require `github.com/enolalabs/sunset@v1.0.1` with `GOPROXY=direct`, import `github.com/enolalabs/sunset/pkg/sunset`, and compile and run a minimal program that calls `sunset.Languages()` and verifies the expected four languages.

This job has repository read permission only. A failure leaves the release in draft state and prevents promotion. Published-release verification-only reruns use the same exact-tag checkout, absolute fixture, clean caches, CLI install smoke test, and external-consumer module test against the existing public release/tag.

### 4.6 Promotion and post-public verification

After draft verification succeeds and a release maintainer approves the protected `release` environment, a dedicated promotion job with `contents: write` changes the draft to a public release. The workflow then performs a lightweight read-only post-public check from `ubuntu-24.04`:

1. Download `checksums.txt` and all five assets through their public release URLs into a fresh directory.
2. Verify the exact public asset set and every SHA-256 digest.
3. Record the public release URL and verification result in the workflow summary.

The post-public check confirms the anonymous download path but is not a substitute for the pre-promotion gate. If it fails because public bytes or availability differ from the verified draft, the workflow fails and maintainers follow the immutable patch-release policy in Section 5; published assets are never silently replaced.

## 5. Failure handling and reproducibility

The release process maintains these invariants:

- A release is never promoted from draft with only a subset of supported targets or before draft verification succeeds.
- Checksums always describe the final downloadable archives, not intermediate executables.
- Version strings, archive names, CLI output, and generated metadata resolve from the same validated tag, whether the binary is packaged with an override or installed from the public module.
- Build and verification jobs cannot publish or modify releases.
- Module preparation never mutates the tagged source tree.
- A workflow rerun may replace assets only while the exact-tag release remains a non-public draft. Once published, reruns enter verification-only mode and cannot replace archives, checksums, notes, or other release metadata.

Once a release is public, its tag, assets, checksums, notes, and metadata are immutable. A post-publication defect is corrected with a new patch tag such as `v1.0.2`; the project does not move the tag or overwrite any part of the published release.

Bit-for-bit reproducibility across different native C toolchains is not a v1.0.1 requirement. The release does require traceable inputs, deterministic naming, native execution, and checksum verification.

### 5.1 Timeouts and pipeline duration

Every job has a finite `timeout-minutes` value:

| Job class | Timeout |
|---|---:|
| Preflight | 30 minutes |
| Each native build and smoke test | 30 minutes |
| Aggregate and draft staging | 15 minutes |
| Draft-asset and public-module verification | 20 minutes |
| Promotion | 10 minutes after environment approval |
| Post-public download verification | 20 minutes |

The workflow summary records the release-state mode, duration of every job, duration of every native matrix target, total job-minutes, active critical-path duration, end-to-end wall-clock duration, and excluded queue/approval time. Total job-minutes is a compute-cost metric and is not compared with the latency target.

The primary latency metric is the active critical-path duration for an `absent` full release. It is the sum of the sequential preflight, aggregate/draft, draft-verification, promotion-execution, and post-public-verification durations plus the longest native matrix-job duration. Runner queue time and protected-environment approval wait are excluded. The initial non-blocking target is at most 30 minutes for this metric under normal GitHub-hosted runner and network conditions.

`draft` resume runs and `published` verification-only runs report the same measurements under their own mode but are not compared with the full-release target and do not contribute samples to its baseline. Exceeding the full-release target emits a visible workflow-summary warning but does not block v1.0.1. Maintainers revisit the target after enough comparable `absent` runs exist to establish a stable baseline.

## 6. Documentation migration

README and CONTRIBUTING will:

- use `github.com/enolalabs/sunset` for clone, install, imports, and links;
- document all five supported targets;
- describe each target as built and verified on its named runner environment, without claiming an untested minimum OS/runtime baseline;
- show `.zip` handling for Windows and `.tar.gz` handling for Linux/macOS;
- avoid combining the moving `/releases/latest/download/` endpoint with a hard-coded versioned filename;
- show a version-explicit download example;
- show checksum-verification commands using `sha256sum` or `shasum -a 256` on POSIX and `Get-FileHash -Algorithm SHA256` on Windows;
- explain that SHA-256 verifies integrity but not publisher authenticity;
- retain the existing CLI and library usage contract.

The repository will contain no canonical-installation example using the old singular organization.

## 7. Testing and acceptance criteria

Phase 1 is complete only when all of the following hold:

### Repository checks

- `go test ./... -race -count=1` passes.
- `go vet ./...` passes.
- `go mod tidy -diff` produces no difference.
- `go list -m` returns `github.com/enolalabs/sunset`.
- No active runtime, release, or user-facing surface defined in Section 4.1 contains `github.com/enolalab/sunset`; historical design/review/audit records are excluded.
- Every action reference in the release workflow is a full commit SHA with an upstream-version comment.
- A repository ruleset protects `v*` tag creation, and the `release` environment requires approval from a designated release maintainer.
- Existing output-schema and snapshot tests remain unchanged unless their only difference is the corrected module identity or version value.

### Workflow checks

- All five targets build on matching native OS/architecture runners.
- Every final archive is extracted on its native runner into a fresh directory, contains exactly the expected root executable, preserves executable permission on POSIX, and passes the complete smoke-test contract from the extracted path.
- Every native job asserts its host OS/architecture and records runner image, Go version, and C compiler identity/version.
- The Windows job resolves and records the preinstalled MinGW-w64 GCC toolchain before building with `CGO_ENABLED=1`.
- README and release notes identify the exact runner environment used to verify each target and do not imply broader runtime compatibility.
- A failed target prevents publication.
- Draft verification and release-environment approval both succeed before promotion.
- Draft and published verification-only jobs checkout the exact tagged commit and use the absolute path to its tracked fixture.
- The release contains exactly five archives and one checksum file.
- Every archive digest matches `checksums.txt` after downloading release assets.
- `sunset version` and generated `index.md` both report `1.0.1` for the v1.0.1 artifacts.
- Release notes explicitly communicate the canonical path migration, lack of shim, target matrix, and checksum instructions.

### Public distribution checks

From a clean Go build and module cache, this command succeeds against the public tag:

```bash
go install github.com/enolalabs/sunset/cmd/sunset@v1.0.1
```

The installed executable passes `sunset version`, `sunset languages`, and `sunset parse --no-cache --quiet --output <temporary-directory> <fixture>`. Both CLI output and generated `index.md` report `1.0.1` through build-info fallback.

From a separate temporary consumer module with clean caches, requiring `github.com/enolalabs/sunset@v1.0.1`, importing `github.com/enolalabs/sunset/pkg/sunset`, and compiling/running a call to `sunset.Languages()` succeeds and returns Go, JavaScript, TypeScript, and Python.

## 8. Explicit non-goals

Phase 1 does not:

- repair incremental index/cache/output consistency;
- remove stale per-file output after source deletion;
- add single-file CLI parsing;
- change quiet-mode or per-file error reporting;
- integrate dependency-graph output;
- change the worker-pool architecture;
- add symlink containment or collision-resistant output paths;
- introduce a coverage threshold;
- sign or notarize release artifacts.

These items remain assigned to Phases 2 through 5 of the improvement program.

## 9. Implementation boundary

Implementation should touch only the files required for canonical identity, dependency metadata, documentation, shared version resolution, release workflow consolidation, and target-specific smoke-test support. It must not opportunistically refactor parser, scanner, cache, engine, analyzer, or output behavior beyond routing existing CLI/output version metadata through the shared resolver.
